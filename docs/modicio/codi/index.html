  1.  trait Base extends AnyRef

    The Base trait defines any class producing or forwarding a Definition compatible specification of Rules.

    The Base trait defines any class producing or forwarding a Definition compatible specification of Rules.

    ConcreteValues are handled the same way as Rules internally.

    As of right now, rules.ConstraintRule and rules.BehaviourRule are not supported, i.e. throw an UnsupportedOperationException.

  2.  abstract class BaseModel extends Fragment with Base

    The BaseModel is a semi-concrete implementation of a Fragment.

    The BaseModel is a semi-concrete implementation of a Fragment. A BaseModel serves as a design-time implementation of a invariant type-fragment which is represented by a concrete implementation of the BaseModel.

    The BaseModel implements the Base trait but does not override its methods. Those must be implemented by concrete hook classes representing concrete types. Not the the methods enforced by the Base tait must provide constant returns. Those must be available at instantiation time.

    A concrete implementation must call the setup method at construction time.

    Additionally, a BaseModel enforces certain behaviour by final overrides: isNode is set to false and the parent set is set to empty, consequently a BaseModel can have no extensions; applyRule and removeRule throw an exception in any case, consequently a BaseModel is invariant.

    See also

    Fragment Base

  3.  class DeepInstance extends AnyRef

    The DeepInstance is the central class of the clabject-hierarchy.

    The DeepInstance is the central class of the clabject-hierarchy. See the external documentation for more information and examples.

    A DeepInstance must have a TypeHandle containing its instantiated type-fragment; A Registry for lazy reloads on unfold() calls, which contains the all instances and fragments used in the sam runtime-environment.

    A DeepInstance is identified by a technical unique instanceId and an identity representing the clabject.

    Following the intended usage, DeepInstances are always constructed using the [[InstanceFactory InstanceFactory]]

    Objectified members of a DeepInstance are bundled by its Shape.

    A DeepInstance serialises to the data-tuple represented by the InstanceData case-class.

    A DeepInstance is a foldable entity. If the DeepInstance is loaded from the Registry or constructed by the InstanceFactory, their extensions are only available as ExtensionData part of the Shape. The client must call unfold() if deep queries and operations should be performed. Only by calling unfold(), the extension hierarchy is loaded from the Registry and wired in form of unfoldedExtensions.

    See also

    InstanceFactory InstanceData Shape

  4.  final class Definition extends Observable with Base

    The Definition class is a final concrete Base implementation representing the set of Rules a Fragment possesses.

    The Definition class is a final concrete Base implementation representing the set of Rules a Fragment possesses.

    The Definition encapsulates the Rules in individual sets and provides an api to edit the rule-set. Note that the Definition class itself does not ensure validity of the resulting model, especially in an extension-hierarchy. If a new Rule is added or an old Rule is removed, the provided DefinitionVerifier is called with the temporary altered rule-set. If the DefinitionVerifier accepts accepts the rule-set as valid, the changes are applied.

    The Observable trait is implemented in this class to enable clients to observe rule-set changes if Rules are added and removed.

  5.  abstract class Fragment extends AnyRef

    The Fragment is the central class of the CoDI type-hierarchy.

    The Fragment is the central class of the CoDI type-hierarchy. It forms the component of the type-composite tree.

    The abstract Fragment contains all concrete functionality to handle associations and rules. Methods regarding extensions are abstract and must be implemented by the Node. Other abstract members must be extended or overwritten by its child classes Node and BaseModel

    Apart from the constructor arguments, the Fragment possess a Definition, a Registry, a DefinitionVerifier and a ModelVerifier.Those classes are not always required but should be set directly after object construction using their respective set-methods.

  9.  abstract class Rule extends AnyRef

    Overall abstract class to represent any rule held by a Definition or BaseModel.

    Overall abstract class to represent any rule held by a Definition or BaseModel.

    Instances of this class and their concrete implementations are stored as RuleData by their serialised value.

    The nativeValue of a Rule must contain all information regarding the Rule and the concrete Rule-type. The nativeValue must follow a definition of the native-language.

    The abstract Rule class handles the id value, which must always be the first value of a nativeValue of any concrete Rule, separated by a ":" from the rest. If no id is known, the nativeValue must start with the ":" directly.

    Definition BaseModel RuleData
