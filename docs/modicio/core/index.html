  2.  class DeepInstance extends AnyRef

    The DeepInstance is the central class of the clabject-hierarchy.

    The DeepInstance is the central class of the clabject-hierarchy. See the external documentation for more information and examples.

    A DeepInstance must have a TypeHandle containing its instantiated type-modelElement; A Registry for lazy reloads on unfold() calls, which contains the all instances and modelElements used in the sam runtime-environment.

    A DeepInstance is identified by a technical unique instanceId and an identity representing the clabject.

    Following the intended usage, DeepInstances are always constructed using the [[InstanceFactory InstanceFactory]]

    Objectified members of a DeepInstance are bundled by its Shape.

    A DeepInstance serialises to the data-tuple represented by the InstanceData case-class.

    A DeepInstance is a foldable entity. If the DeepInstance is loaded from the Registry or constructed by the InstanceFactory, their parentRelations are only available as ParentRelationData part of the Shape. The client must call unfold() if deep queries and operations should be performed. Only by calling unfold(), the parentRelation hierarchy is loaded from the Registry and wired in form of unfoldedParentRelations.

    See also

    InstanceFactory InstanceData Shape

  3.  class Definition extends Observable with Base

    The Definition class is a concrete Base implementation representing the set of Rules a ModelElement possesses.

    The Definition class is a concrete Base implementation representing the set of Rules a ModelElement possesses.

    The Definition encapsulates the Rules in individual sets and provides an api to edit the rule-set. Note that the Definition class itself does not ensure validity of the resulting model, especially in an parentRelation-hierarchy. If a new Rule is added or an old Rule is removed, the provided DefinitionVerifier is called with the temporary altered rule-set. If the DefinitionVerifier accepts accepts the rule-set as valid, the changes are applied.

    The Observable trait is implemented in this class to enable clients to observe rule-set changes if Rules are added and removed.

  6.  class ModelElement extends AnyRef

    The ModelElement is the central class of the CoDI type-hierarchy.

    The ModelElement is the central class of the CoDI type-hierarchy. It forms the component of the type-composite tree.

    The ModelElement contains all concrete functionality to handle associations, rules and parentRelations.

    Apart from the constructor arguments, the ModelElement possess a Definition, a Registry, a DefinitionVerifier and a ModelVerifier.Those classes are not always required but should be set directly after object construction using their respective set-methods.

    Overall abstract class to represent any rule held by a Definition.

    Overall abstract class to represent any rule held by a Definition.

    Instances of this class and their concrete implementations are stored as RuleData by their serialised value.

    The nativeValue of a Rule must contain all information regarding the Rule and the concrete Rule-type. The nativeValue must follow a definition of the native-language.

    The abstract Rule class handles the id value, which must always be the first value of a nativeValue of any concrete Rule, separated by a ":" from the rest. If no id is known, the nativeValue must start with the ":" directly.
